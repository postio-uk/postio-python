# postio-python — Claude Code working notes

Python SDK for `postio-api`. Mirrors `@postio/core` (the JS family's
runtime client) with idiomatic Python ergonomics. Lives in its own repo
because Python's toolchain (uv/pip/PEP 621/Pydantic) doesn't co-exist
cleanly with the `postio-integrations` pnpm workspace.

Read [`README.md`](./README.md) for the customer-facing surface; this file
is the operational guide.

## Stack

- **Python 3.10+**. Uses union-type syntax (`str | None`), pattern matching,
  and Pydantic v2 strict mode — drop floor only when 3.10 EOLs.
- **uv** for project management, builds, and CI. `pyproject.toml` is
  PEP 621 standard; `hatchling` is the build backend.
- **httpx** for HTTP (sync + async, same surface).
- **Pydantic v2** for response models — generated by `datamodel-code-generator`
  (NOT `openapi-python-client`; see decision below).

## Why datamodel-code-generator and not openapi-python-client

The original brief recommended `openapi-python-client` (generates a full
client we'd then wrap). After looking at the spec — 6 endpoints, 15
schemas — the wrap-a-generated-client pattern adds plumbing we don't need.

The cleaner mirror of the JS pattern is:

| JS family | Python |
|---|---|
| `@postio/api-types` (types only, generated from OpenAPI) | `postio/_models.py` (generated by datamodel-code-generator) |
| `@postio/core` (runtime, hand-written) | `postio/client.py` (hand-written) |

One generator, one client, one HTTP transport. If the SDK ever needs to
expose every endpoint mechanically (e.g. for an admin namespace), revisit.

## Layout

```
postio-python/
├── postio/
│   ├── __init__.py        re-exports public surface
│   ├── _version.py        single source of truth for the version
│   ├── _models.py         generated by scripts/codegen.py — see "Spec drift" below
│   ├── _http.py           shared httpx plumbing, retry config, envelope parsing
│   ├── client.py          PostioClient (sync) + AsyncPostioClient (async)
│   ├── exceptions.py      typed exception hierarchy
│   └── py.typed           PEP 561 marker
├── scripts/codegen.py     fetch spec → regenerate _models.py
├── tests/
│   ├── conftest.py        loads umbrella .env, picks stage vs prod URL by key
│   ├── test_offline.py    respx-mocked, no network or key — runs on every PR
│   ├── test_address.py    live tests (skipped if no key)
│   ├── test_email.py
│   ├── test_phone.py
│   └── test_connect.py
└── .github/workflows/
    ├── ci.yml             ruff + mypy + offline pytest matrix on every push
    └── release.yml        on v* tag → build + publish to PyPI via OIDC
```

## Common commands

```bash
# install everything
uv sync

# run all tests (offline + live; live skipped if no key in env)
uv run pytest

# offline-only
uv run pytest tests/test_offline.py

# lint + format + type
uv run ruff check postio/ tests/
uv run ruff format postio/ tests/
uv run mypy postio/

# regenerate models from live spec
uv run python scripts/codegen.py
```

Local dev picks up the umbrella `~/PROJECTS/ONNO/POSTIO/.env`
automatically (via python-dotenv in `tests/conftest.py`). When
`POSTIO_API_KEY_STAGE` is set, tests hit `stage-api.postio.co.uk`. When
only `POSTIO_API_KEY_PROD` is set, they hit `api.postio.co.uk`. Live
tests are skipped if neither is present.

## Branch + deploy model

- `stage` — working branch, mirrors the rest of the org.
- `master` — push triggers the `live-test` job (real stage credit) and is
  the gate for publishing. Releases are cut by tagging `vX.Y.Z` on master.
- `release.yml` is **idempotent**: if `postio==<version>` is already on
  PyPI, the workflow exits cleanly without republishing.

## Releasing

```bash
# 1. bump version
edit postio/_version.py        # __version__ = "0.1.1"
edit pyproject.toml            # version = "0.1.1"
edit CHANGELOG.md              # move Unreleased → 0.1.1, add release notes

# 2. commit + tag + push
git add -A && git commit -m "release: v0.1.1"
git tag v0.1.1
git push origin master --tags
```

PyPI publish flows through GitHub Actions Trusted Publishing (OIDC) —
no long-lived `PYPI_API_TOKEN` secret. The publisher is registered at
PyPI for `(postio-uk/postio-python, release.yml, environment: pypi)`.

## Versioning

Independent SemVer; **not** lockstep with `@postio/openapi`. The spec
version lives in the response envelope and is independent of SDK
ergonomics changes. Stay on `0.x` until the API contract is committed
stable; bump to `1.0` only with a real public-API freeze.

## Spec drift

`postio/_models.py` is generated, but a couple of fields need manual
patches to reflect runtime reality:

1. **`PhoneResult`** — every nullable field is patched to `= None`
   default. The spec marks them `required` with type `[string, null]`,
   but on invalid input the live API drops them entirely.
2. **`PhoneResult.isReachable`** — patched to `bool | str | None`. Spec
   says string-only; live API returns `bool`.

`scripts/codegen.py` prints a reminder of these patches on every regen.
When postio-api ships a spec/runtime alignment, drop the patches and
let the generator drive `_models.py` verbatim.

## Secrets the CI needs

| Secret | Used by | Notes |
|---|---|---|
| `POSTIO_API_KEY_STAGE` | `ci.yml` (live-test job) | A `pk_*` key in stage KV. Must include `localhost` in `allowedUrls` (or be unrestricted) so GH-hosted runners can call the API. |

PyPI publishing uses **Trusted Publishers (OIDC)** — no PyPI token in
GH secrets. Configured at:

> https://pypi.org/manage/project/postio/settings/publishing/

Bound to `(postio-uk/postio-python, release.yml, environment: pypi)`.
